# 第9章：X Developer設定で不安になった話

## 1. 用語の洪水

100ドルを支払い、意を決してX Developer Portalにアクセスしました。
そこには、無機質な英語のダッシュボードが広がっていました。

「Projectを作れ」「Appを作れ」
そこまでは分かります。
しかし、"Keys and Tokens" のタブを開いた瞬間、私は固まりました。

*   **API Key**
*   **API Key Secret**
*   **Bearer Token**
*   **Client ID**
*   **Client Secret**
*   **Access Token**
*   **Access Token Secret**

「……多すぎる」
どれが必要なのか？ 全部必要なのか？
そもそも、API Key と Client ID は何が違うのか？
Access Token は私のものなのか、Botのものなのか？

ドキュメントを読んでも、"OAuth 1.0a" と "OAuth 2.0" の違いについて延々と書かれており、
「で、結局どれを使えば単純な投稿ができるの？」というシンプルな問いの答えが見つかりません。

## 2. 間違えることへの恐怖

さらに私を萎縮させたのは、**「Regenerate（再生成）」**ボタンの重さです。
一度表示されたSecretは、二度と表示されません。
「これをコピーし忘れたらどうなるんだろう？」
「間違ってRegenerateしてしまったら、既存の連携が壊れるんじゃないか？」

たかが文字の羅列ですが、それが「お金（100ドル）を払った権利」へのアクセスキーだと思うと、手が震えます。
設定を間違えてアカウントがBANされたりしないだろうか。
そんな不安が頭をよぎり、設定画面の前でクリックできずにいました。

## 3. Antigravityに画面を見せる

私は、恥を忍んでAntigravityに助けを求めました。
画面のスクリーンショットを撮り（機密部分は黒塗りで隠して）、そのままAntigravityに貼り付けました。

「これ、どれが何？ 私が自動投稿Botを作るには、どれを保存しておけばいいの？」

すると、Antigravityはまるで熟練のガイドのように、画像の上に赤枠をつけて返してくれました。

> 「落ち着いてください。あなたが今回使うのは、この4つだけです。
>
> 1.  **API Key** (ユーザー名のようなもの)
> 2.  **API Key Secret** (パスワードのようなもの)
> 3.  **Access Token** (Botとしての入場券)
> 4.  **Access Token Secret** (Botとしての署名)
>
> Client IDやBearer Tokenは、今回は使いません。無視して大丈夫です。
> また、Settingsタブで **"Read and Write"** 権限にチェックを入れるのを忘れないでください。
> デフォルトでは "Read Only" になっていて、投稿しようとするとエラーになります」

## 4. 霧が晴れる

「なんだ、これだけでいいのか」
Antigravityの言葉で、複雑怪奇に見えた画面が、急にシンプルな入力フォームに見えてきました。

私は言われた通り、4つの文字列をコピーし、メモ帳……ではなく、安全な場所に保管する準備を始めました。
ここでもまた、エンジニアとしての「セキュリティへの懸念」が顔を出したからです。

## 到達状態 (Check Point)

*   [ ] X Developer Portalの複雑な用語（Key/Secret/Token）に圧倒された経験がある。
*   [ ] スクリーンショットを使ってAIに画面のガイドをさせた。
*   [ ] 必要なのは「4つのキー」と「Read/Write権限」だけだと理解した。

キーは手に入りました。
しかし、これをコードに直書きするのは御法度です。
次章、最後の関門である「セキュリティと鍵管理」についてお話しします。
