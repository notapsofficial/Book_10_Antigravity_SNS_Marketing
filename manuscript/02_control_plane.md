# Part 2｜Antigravityを制御層として使う

## 1. 概念：クラウドインフラとしてのSNS運用

### Data Plane（実行層）と Control Plane（制御層）
AWSやKubernetesを知っているなら、この概念は直感的に理解できるはずだ。
SNS運用を、インフラストラクチャとして捉え直そう。

*   **Data Plane (実行層)**:
    *   **場所**: X (Twitter), LinkedIn, Instagram などのプラットフォーム。
    *   **流れるもの**: 実際の投稿（パケット）、いいね、リプライ、トラフィック。
    *   **性質**: 動きが速く、ノイズが多く、消耗する泥臭い現場。これまでのあなたはここにいた。

*   **Control Plane (制御層)**:
    *   **場所**: AIエージェント（Antigravity）、あなたのローカル環境、API管理画面。
    *   **流れるもの**: ポリシー、ルール、設定、承認シグナル。
    *   **性質**: 静的で、論理的で、抽象度が高い司令室。これからのあなたはここに座る。

あなたがやるべきことは、泥まみれになって現場（タイムライン）で叫ぶことではない。
司令室から**「こういうトーンで、こういう層に向けて情報を供給せよ」**というYAMLファイルを適用（Apply）することだ。

### 「編集者」という権限
出版社を想像してほしい。
あなたは「ライター」ではない。「編集長」だ。
ライター（AI）が書いてきた原稿に対して、
「これじゃ読者が飽きる。もっとパンチの効いた見出しにして」
「この表現は法的リスクがあるからボツ」
そう指示を出すのが役割だ。

自分で原稿用紙を埋めようとするバカな編集長はいない。
編集長の仕事は「企画（方向性）」と「承認（品質担保）」の二点に集約される。

## 2. 実践：プロンプト＝仕様書

Antigravity において、最も重要な資産はこの「プロンプト」だ。
これはAIへのチャットではない。**あなたの分身の定義ファイル**だ。

### 悪いプロンプトの例（解像度が低い）
```text
エンジニア向けの役立つ情報をツイートして。
バズる感じでお願い。
```
これでは、AIは「当たり障りのない一般論」しか返さない。
「プログラミング学習では継続が大事です！ #駆け出しエンジニアと繋がりたい」
こんなゴミのようなポストを量産されても、あなたのブランド（信頼）は毀損するだけだ。

### 良いプロンプトの例（仕様が明確）
以下のように、構造化して記述する。これをテンプレートとして保存しておくのだ。

```markdown
# Role Definition
あなたは「歴15年のシニアバックエンドエンジニア」だ。
これまでに数々のデスマーチを生き延び、技術的負債と戦ってきた歴戦の勇者である。
最近のキラキラした表面的な技術トレンドには懐疑的だが、
基本技術（アルゴリズム、ネットワーク、OS）の重要性は痛感している。

# Target Audience
*   3-5年目のミドルクラスのエンジニア
*   技術選定に迷っているテックリード
*   「このままでいいのか」と不安なSIer出身のエンジニア

# Tone & Style (Critical)
*   **断定調**: 「〜だと思います」ではなく「〜だ」「〜である」と言い切る。
*   **非媚び**: 読者に迎合しない。耳の痛いことでもズバリと言う。
*   **具体性**: 抽象論で終わらせず、具体的な技術名（Redis, Rust, Linuxなど）を例示に出す。
*   **禁止事項**: 絵文字の多用、ハッシュタグの乱用、「頑張りましょう」系の精神論。

# Task
上記のペルソナに基づき、以下のトピックについて140字以内の投稿案を3つ作成せよ。
トピック：【マイクロサービス化の失敗事例について】
```

このレベルまで定義して初めて、AIは「あなたらしい（あるいは、あなたが演じたい）」投稿を生成できる。

## 3. アジャイルな運用：ルールをデプロイする

### ターゲットを変える＝設定を変える
運用を始めて1ヶ月後、「なんかつまらないな」と思うかもしれない。
あるいは反応が鈍化するかもしれない。
そんな時、手動運用の人は「もっと面白いことを書かなきゃ」と焦り、無理をする。

Antigravity の運用者は違う。
「ターゲット設定が『シニア』すぎて狭かったか。よし、『駆け出し』向けにチューニングし直そう」
そう考えて、プロンプトの `# Target Audience` を書き換える。
あるいは、「少し毒舌すぎたから、もう少し親しみやすくしよう」と `# Tone` を修正する。

これが**「ルールをデプロイする」**という感覚だ。
あなたは個別のツイートを直すのではなく、生成の源流にある**「金型」**を修正する。
金型を直せば、そこから生まれる製品（投稿）はこれ以降すべて修正後の形になる。
これこそが「レバレッジ」だ。

### システムオーナーの責任
AIに生成させると、たまにトンチンカンなことを言う。
それを「AIが勝手にやった」と言い訳してはいけない。
それは**「あなたの指示（プロンプト）が甘かった」**というバグだ。

*   AIが嘘をついた → `# Rules` に「事実は必ず検索して裏を取れ」と書いていたか？
*   AIが失礼なことを言った → `# Constraints` に「攻撃的な表現は禁止」と明記していたか？

プロンプトを書くということは、AIという部下の教育係になるということだ。
部下のミスは上司の責任。
だが、一度教育が行き届けば、部下はあなたの何倍もの速度で正確に仕事をこなし続ける。
