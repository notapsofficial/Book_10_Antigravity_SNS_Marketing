# 第12章：初めてSuccessレスポンスを見た瞬間

## 1. 201 Created

ペアプログラミングのようなデバッグを経て、ついにその瞬間は訪れました。
私のターミナルに、美しいレスポンスが表示されました。

```
Response Code: 201
ID: 1734567890123456789
Text: Hello, world from Antigravity. #Flutter #Python
```

「201 Created」。
HTTPステータスコードにおいて、これほど安堵する数字はありません。
それは、私の悩み（認証、権限、コードのバグ）がすべて解消され、リクエストが正しく受理されたことの証明です。

## 2. アプリを開かずに確認する

以前の私なら、ここで慌ててスマホを取り出し、Xアプリを開いて投稿を確認していたでしょう。
しかし、今回の私は冷静でした。
というか、そもそも「アプリを開きたくない」からこのシステムを作ったのです。

私はブラウザで、自分のプロフィールのURLを直接叩きました。
`twitter.com/my_username`

最新の投稿欄に、あの一文が表示されています。
「Hello, world from Antigravity.」

タイムラインの喧騒も、おすすめタブの誘惑も目に入りません。
ただ、自分が意図したアウトプットだけが、静かにそこに存在していました。

「できた……」

## 3. 「一方通行」の心地よさ

この瞬間、私は奇妙な心地よさを感じていました。
今までSNS投稿といえば、賑やかな広場（アプリ）に飛び込み、周りの声に圧倒されながら、大声を張り上げるような行為でした。

しかし、API経由での投稿は違います。
静かな自室（ターミナル）から、手紙をポストに投函するような感覚です。
投函した手紙がどう読まれるか、周りがどう騒いでいるか、今の私には関係ありません。

「これだ。私が欲しかったのは、この静けさだ」

## 4. 成功体験の質が変わった

以前の「成功」は、「いいねがたくさんついた」とか「バズった」という他者評価依存のものでした。
しかし、今回の成功は違います。
「自分で設計したシステムが、意図通りに動いた」という、純粋なエンジニアリングとしての成功です。

この小さな「201」の成功体験が、私のSNSに対する苦手意識を払拭してくれました。
SNSは「怖い場所」から、「制御可能なI/O（入出力）先」へと変わったのです。

## 到達状態 (Check Point)

*   [ ] HTTP 201 (Created) のレスポンスを確認した。
*   [ ] アプリを開かず、ブラウザやURL指定で投稿を確認できた。
*   [ ] SNS運用が「騒がしい行為」から「静かな送信作業」に変わった感覚を味わった。

さて、パイプラインは繋がりました。
ここからは運用フェーズです。
私はこのシステムを使って、どのように「書かないSNS」を実践し始めたのか。
そこで重要になるのが、私の別の著書でも触れた「資産型」の考え方です。
